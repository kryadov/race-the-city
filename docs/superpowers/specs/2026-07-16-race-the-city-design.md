# Race the City — дизайн-документ

**Дата:** 2026-07-16
**Статус:** черновик на вычитку

## 1. Идея и цель

3D-игра в браузере, где можно свободно кататься на машине по реальному городу.
Геометрия города (дороги, здания) генерируется из открытых данных **OpenStreetMap**,
а рельеф земли — из открытых тайлов высот (**Terrain Tiles**).

**Цель проекта:** хобби / для себя, поучиться. Отсюда приоритеты — удобный
итеративный стек, быстрый прогресс, понятный код, из которого ясно «как это устроено».
Масштабируемость и монетизация не важны.

**Ощущение:** свободный круиз. Вводишь любой город → грузится кусок вокруг точки →
катаешься и кайфуешь от «вот мой район». Никаких соперников, чекпоинтов и таймеров.

## 2. Ключевые решения

| Тема | Решение | Почему |
|------|---------|--------|
| Стек | Vite + TypeScript + Three.js | Быстрая итерация, много готовых примеров OSM+3D, типы помогают с сырыми данными, учебная ценность |
| Бэкенд | Нет, статичный сайт | «Просто открыл html», шарится ссылкой, деплой на GitHub Pages |
| Геймплей | Свободная езда | Простейший MVP: нет AI, целей, таймеров — фокус на мире и езде |
| Охват | Любой город по вводу, кусок ~1 км радиусом | Вау-эффект «вбей свой город», но объём данных вменяемый |
| Данные города | Nominatim (геокод) + Overpass API (дороги/здания), кэш в IndexedDB | Ноль бэкенда; кэш обходит повторные запросы и лимиты |
| Рельеф | Terrain Tiles (формат Terrarium, AWS), абстракция `ElevationProvider` | OSM НЕ содержит рельеф; Terrarium бесплатен и без токена; фолбэк на плоскость |
| Физика | Аркада + свой лёгкий модуль столкновений | Под аркаду физдвижок избыточен; свой код = учебная ценность и полный контроль |
| Столкновения | Круг↔полигон в плоскости XZ + пространственная сетка | Сквозь дома нельзя; сетка = проверяем только соседние здания |
| Стиль | Low-poly (закрашенные объёмы), wireframe-режим позже | Красиво, быстро, отличная производительность из одних и тех же данных |

## 3. Что важно знать про источники данных

**Высота ЗДАНИЙ** — есть в OSM частично: теги `height` (метры) или `building:levels`
(этажи). Заполнены не всегда: есть тег — берём, нет — дефолт по типу здания.

**Высота РЕЛЬЕФА** (склоны, холмы) — в OSM НЕТ. OSM — 2D-векторные данные, у узлов
только `lat/lon`. Поэтому рельеф берём из отдельного источника — DEM-тайлов Terrarium:
высота закодирована в пикселе, `height = (R*256 + G + B/256) - 32768` (метры).

Итог: **OSM даёт дороги + контуры зданий, Terrain Tiles — рельеф**. Два источника
совмещаются по координатам.

## 4. Архитектура

Главный принцип — разделить **«получить и понять данные»** и **«нарисовать 3D»**.
Гео/террейн-слои не знают о Three.js (отдают чистые числа); мир-строитель не знает о сети.
Это даёт тестируемость и чистые границы.

```
src/
  geo/              — данные OSM (без Three.js, без рендера)
    geocode.ts        Nominatim: "Тбилиси" → {lat, lon}
    overpass.ts       строим запрос по bbox, фетчим дороги + здания
    cache.ts          IndexedDB: ключ = bbox, чтобы не дёргать повторно
    parse.ts          сырой OSM JSON → {roads[], buildings[]} с тегами
    project.ts        lat/lon → локальные метры (плоская проекция вокруг центра)

  terrain/          — рельеф (без Three.js)
    provider.ts       интерфейс ElevationProvider.heightAt(x, z)
    terrarium.ts      грузим/декодим Terrain-тайлы → высоты
    heightmap.ts      строим продавленную сетку-землю
    flat.ts           фолбэк-провайдер: высота = 0

  world/            — превращение данных в меши (без сети)
    ground.ts         сетка-земля по heightmap
    roads.ts          полилинии → плоские «ленты», ширина по типу, драпировка по рельефу
    buildings.ts      контур + высота → экструзия в коробку; отдаёт меш И полигон для физики
                      (фундамент на уровне земли в своей точке)

  physics/          — столкновения (чистая математика, без Three.js)
    grid.ts           пространственная сетка: bbox зданий → клетки, запрос соседей
    collide.ts        круг↔полигон: выталкивание наружу + скольжение вдоль стены

  vehicle/          — машина
    car.ts            аркадный контроллер (газ/тормоз/поворот, инерция);
                      после движения → collide.resolve(); Y следует за рельефом
    input.ts          клавиатура WASD/стрелки

  app/
    scene.ts          сцена, свет, туман, камера-преследователь
    loop.ts           главный цикл
    main.ts           склейка всего

  ui/
    cityInput.ts      поле ввода города + кнопка «Поехали»
    loading.ts        индикатор загрузки

test/                 — юнит-тесты чистых функций
index.html
```

## 5. Поток данных

```
ввод города
  → geocode (Nominatim)             название → {lat, lon}
  → строим bbox радиусом R          (старт R ≈ 1 км)
  → cache.get(bbox)?
       есть → берём из IndexedDB
       нет  → overpass.fetch        дороги + контуры зданий
            → cache.put
  → parse                           OSM JSON → roads[] / buildings[]
  → project                         lat/lon → метры (центр = 0,0)
  → terrain: грузим Terrain-тайлы для bbox → ElevationProvider
  → world.build                     меши земли/дорог/зданий поверх ElevationProvider;
                                     полигоны зданий → physics.grid
  → спавним машину, ставим на землю
  → едем

каждый кадр:
  input → car двигает позицию (газ/руль/инерция)
        → collide.resolve(позиция, скорость, grid)   выталкивает из домов
        → Y = ElevationProvider.heightAt(x, z)        посадка на рельеф
        → камера-преследователь, рендер
```

## 6. Объём MVP

Первая рабочая версия:
- Поле ввода: город/координаты → загрузка области ~1 км.
- **Рельеф** из Terrain Tiles (не-плоская земля); фолбэк на плоскую при сбое тайлов.
- **Дороги**-ленты (ширина по типу: шоссе шире, дворовый проезд уже), драпированы по рельефу.
- **Здания**-коробки (low-poly, плоские цвета), высота из `height`/`building:levels`
  или дефолт; фундамент на уровне земли.
- **Аркадная машинка**: WASD/стрелки, приятная инерция, камера сзади, Y следует за рельефом.
- **Столкновения со зданиями**: сквозь дома нельзя, вдоль стены — скользим.
- Индикатор загрузки + обработка ошибок.

**Явно НЕ в MVP** (потом): wireframe-переключатель, парки/вода/деревья, полуреалистичная
физика (занос/сцепление, → возможен переезд на Rapier), миникарта, день/ночь, звук,
соперники/гонки/таймеры.

## 7. Обработка ошибок

- Nominatim не нашёл город → сообщение «город не найден, попробуй иначе».
- Overpass таймаут/лимит → понятная ошибка + «повторить»; подсказка уменьшить радиус.
- Terrain-тайлы не пришли → фолбэк на плоскую землю (`flat.ts`), игра не ломается.
- Битая/пустая геометрия здания → пропускаем один объект, сборку не роняем.
- Оффлайн → если bbox есть в кэше OSM, работаем без сети (рельеф — по доступности тайлов).

## 8. Тестирование

Юнит-тесты на **чистые функции** (без рендера и сети):
- `project`: известные lat/lon → ожидаемые метры.
- `overpass`: bbox → корректная строка запроса.
- `terrarium`: декодирование RGB пикселя → правильная высота.
- `parse` / `buildings`: на маленьком зафиксированном OSM-JSON — верные контуры и высоты,
  битые объекты отсеиваются.
- `grid`: здания раскладываются по правильным клеткам; запрос соседей корректен.
- `collide`: машина у стены выталкивается наружу; скорость вдоль стены сохраняется,
  в стену — гасится; угол дома не «залипает».

Рендер и «ощущение от езды» проверяем руками/глазами — это нормально для игры.

## 9. Возможное развитие (за пределами MVP)

- Wireframe/неон-режим переключателем (данные те же).
- Парки, вода, деревья из OSM (`leisure=park`, `natural=water` и т.п.).
- Полуреалистичная физика (занос, сцепление, масса) — при желании переход на Rapier.
- Больший охват / подгрузка соседних кусков.
- Миникарта, день/ночь, звук двигателя.
- Гоночный режим поверх: чекпоинты, таймер, лучшее время.
